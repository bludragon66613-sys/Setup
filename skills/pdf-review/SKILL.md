# PDF Review

A workflow skill that turns Claude's multimodal Read tool into a PDF
QA reviewer. The skill exists because PDF generation pipelines (puppeteer,
pdf-lib, React-PDF, Pillow stitching, etc.) all introduce subtle
rendering issues -- alignment drift, font substitution, page-break
splits, color-profile mismatches, broken layouts, missing assets --
that look fine in the source HTML or design doc but break in the
final PDF. A human review catches them; this skill makes Claude do
the same review systematically.

## When to use

Apply this skill when:

- A PDF was just generated by any tool (puppeteer print, pdf-lib,
  React-PDF, Pillow stitching, LaTeX, Word export, etc.)
- A PDF is about to be shared with a stakeholder, client, or external
  audience and you need a confidence check
- A PDF rebuild is being compared against a previous version (regression
  check)
- A PDF is being prepared for print and needs a final visual pass
- Any time the user says "review this PDF", "check this PDF for issues",
  "is this PDF ready", or similar

## When NOT to use

Skip this skill when:

- The user wants OCR / text extraction (use `pypdf` or `pymupdf`
  directly instead)
- The PDF is text-only and the user wants a content audit (use Read
  on the source markdown / docx instead)
- The PDF is a scientific paper or reference document with no design
  concerns
- The user is debugging a PDF *generator* (the bug is in the code,
  not the output -- this skill reviews output)

## Prerequisites

Required Python packages (all standard, install if missing):

```bash
pip install pymupdf pillow
```

`pymupdf` (imported as `fitz`) renders PDF pages to images. `Pillow`
handles image manipulation if needed. No external binaries (no
poppler, no pdftoppm, no Ghostscript) -- pure Python.

## Workflow

### Step 1: Render every page to PNG

Render the PDF to a temporary directory at sufficient zoom for
visual inspection. 1.5x is usually enough; 2x for fine-detail QA.

```python
import fitz, os
from pathlib import Path

PDF_PATH = "/path/to/input.pdf"
OUT_DIR = Path.home() / "pdf-review-temp"
OUT_DIR.mkdir(exist_ok=True)

doc = fitz.open(PDF_PATH)
zoom = 1.5
mat = fitz.Matrix(zoom, zoom)
for i in range(doc.page_count):
    page = doc[i]
    pix = page.get_pixmap(matrix=mat)
    pix.save(OUT_DIR / f"page-{i+1:03d}.png")
print(f"rendered {doc.page_count} pages to {OUT_DIR}")
```

### Step 2: Walk every page with the Read tool

Use the Read tool on each `page-NNN.png`. Claude's multimodal Read
treats PNGs as visible images.

For PDFs with up to ~15 pages, read every page. For longer PDFs,
sample strategically:

- Always read: page 1 (cover), page 2 (TOC if present), the last page
- Always read: any page the user explicitly flagged
- Sample: every Nth page so you cover the document evenly
- Read more densely around any flagged issue

### Step 3: Score each page on the 6-dimension rubric

For every page you read, evaluate against these six dimensions and
score each on a `pass / minor / major / critical` scale:

| Dimension | What to look for |
|---|---|
| **Alignment** | Margins consistent? Elements aligned to a grid? Text not running off the edge? Images not bleeding into other elements? |
| **Typography** | Font rendering correct (no fallback / substitution)? Line breaks readable? Hyphenation sensible? Letter spacing intact? Kerning correct? Heading hierarchy clear? |
| **Color** | Brand colors accurate (not desaturated, not gamut-shifted)? Backgrounds consistent? Contrast meets accessibility needs? No unintended color bleeds? |
| **Layout** | Page breaks land in sensible places (not mid-paragraph, mid-card, mid-table)? White space balanced? Visual hierarchy clear? Cards / sections internally cohesive? |
| **Content** | Text readable at intended viewing size? Images not pixelated? Captions present where expected? Footers / page numbers present and correct? Placeholder text removed? |
| **Brand fidelity** | Does the page MATCH the brand reference (logo, color, type, voice)? Tokens applied correctly? Logo positioned correctly? Sign-offs / vanity marks present where expected? |

### Step 4: Produce the report

Output a structured markdown report with this format:

```markdown
# PDF Review: [filename]

**Reviewed:** YYYY-MM-DD
**Pages:** N
**Pages inspected:** M of N
**Overall verdict:** PASS / MINOR ISSUES / MAJOR ISSUES / CRITICAL

## Summary

One paragraph describing the overall state of the PDF.
What works well. What is broken. Whether this PDF is shippable.

## Findings by severity

### CRITICAL (blocks shipping)

- **[Page X, dimension]** — short description of the issue, why it
  is critical, suggested fix
- ...

### MAJOR (should fix before sharing)

- ...

### MINOR (nice to fix)

- ...

### PASS (no issues)

- ...

## Per-page notes

| Page | Alignment | Typography | Color | Layout | Content | Brand | Notes |
|---|---|---|---|---|---|---|---|
| 1 | ✅ | ✅ | ✅ | ✅ | ✅ | ✅ | Clean cover |
| 2 | ✅ | ⚠️ | ✅ | ❌ | ✅ | ✅ | Heading wraps awkwardly; card 3 splits across page |
| ... |

Legend: ✅ pass · ⚠️ minor · 🟧 major · ❌ critical

## Recommendations

Numbered, actionable list of fixes ordered by severity.

## Sign-off

If verdict is PASS or MINOR ISSUES: "Ready to ship."
If verdict is MAJOR or CRITICAL: "Not ready. Fix [X] then re-review."
```

### Step 5 (optional): Regression compare

If a `before.pdf` is provided alongside the `after.pdf`:

1. Render both PDFs to PNG sets in separate temp dirs
2. For each page that exists in both, read the before and after
   versions back-to-back via the Read tool
3. Note any visible regressions in the report under a new section:
   "## Regressions vs previous version"
4. Highlight any improvements as well -- regression checks should
   capture both directions

## Important rules

1. **Never claim a PDF is shippable without visually reading at least
   the cover, TOC (if any), every section divider, and the last page.**
   A page-count check is not a review.

2. **Trust your eyes, not the source code.** If the source markdown
   says "this looks great" and the rendered PDF looks broken, the PDF
   is broken. The source is irrelevant to the review verdict.

3. **Surface the cause when you can.** Many PDF issues have predictable
   root causes -- font substitution from missing webfonts, page-break
   issues from missing CSS print rules, color drift from RGB→CMYK
   conversion. If you recognize a pattern, mention the likely cause
   in the report so the user can fix the generator instead of patching
   the output.

4. **Keep severity proportional.** A misaligned footer is not the same
   as a missing page. Critical means "would embarrass the user if
   sent to a stakeholder." Minor means "perfectionist nitpick." Be
   honest about which you are flagging.

5. **Clean up the temp render dir when done** unless the user asks
   you to keep it.

## Common failure modes the skill catches

- **Font fallback / substitution** — typography dimension. Cause:
  webfonts not loaded before render. Fix: wait for `document.fonts.ready`
  in puppeteer, or embed fonts directly in the PDF generator.
- **Page-break splits** — layout dimension. Cause: missing `page-break-
  inside: avoid` CSS rule. Fix: add print CSS or switch to a generator
  that respects page boundaries.
- **Color drift** — color dimension. Cause: RGB→CMYK conversion or
  color-profile mismatch. Fix: lock color profile in the generator.
- **Missing images** — content dimension. Cause: relative paths broken
  in the print context. Fix: use absolute paths or embed as data URIs.
- **Mobile viewport leakage** — alignment dimension. Cause: generator
  was set to a mobile viewport but the PDF is meant for desktop.
  Fix: set viewport to match intended page size.
- **Watermark / placeholder text** — content dimension. Cause:
  unfinished content shipped. Fix: search the source for "Lorem", "TODO",
  "TBD" before regenerating.
